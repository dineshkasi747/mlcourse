Complete EDA workflow — what to do after the report

Automated tools give you the first pass. This 7-step workflow shows what to do with the findings.

1
Run the automated tool first
One line gives you 80% of what you need. Read every alert — they flag the most important issues.
from ydata_profiling import ProfileReport report = ProfileReport(df, title="My Dataset") report.to_notebook_iframe()
2
Fix data types
Cast low-cardinality strings to category, parse date strings, convert ID columns to str, downcast int64 to int32.
for col in df.select_dtypes('object'): if df[col].nunique() / len(df) < 0.05: df[col] = df[col].astype('category')
3
Handle missing values
Check if missing is random (MCAR), conditional (MAR), or informative (MNAR). Each requires a different strategy.
df.isnull().sum().sort_values(ascending=False) / len(df) # > 80% missing: consider dropping the feature
4
Investigate outliers
Do NOT auto-remove outliers. Understand why they exist. Use IQR method to flag them, then decide case by case.
q1, q3 = df[col].quantile([0.25, 0.75]) iqr = q3 - q1 outliers = df[(df[col] < q1-3*iqr)|(df[col] > q3+3*iqr)]
5
Fix high correlations (multicollinearity)
Drop one of any pair with |r| > 0.85, or use PCA/Ridge to handle it in the model.
corr = df.corr().abs() high = corr[corr > 0.85].stack().reset_index() high = high[high['level_0'] != high['level_1']]
6
Check class balance (for classification)
Check if the target variable is imbalanced. If the minority class is <20%, plan your resampling strategy now.
df['target'].value_counts(normalize=True) * 100
7
Document your findings
Write a short EDA summary before moving to modelling. What did you find? What did you fix? What risks remain?
# EDA Summary template # Rows: X Cols: Y Missing: Z% # Key findings: ... # Actions taken: ... # Risks: ...

Common mistakes

x
Running ydata-profiling on a 5-million-row dataset. It will run for hours or crash. Use minimal=True mode, or sample 10,000 rows first: df.sample(10000).profile_report().
x
Skipping EDA because "the tool generated a report". Automated reports flag issues but do not fix them. You still need to act on every alert — the tool just speeds up the discovery phase.
x
Not reading the Alerts / Warnings tab. This is the most valuable section. It flags duplicates, high cardinality, zero variance, high correlation, skew, and missing values. Read it first.
x
Running profiling on the full dataset before train-test split. If you use insights from the test set to make preprocessing decisions, that is data leakage. Profile only the training set for decisions that affect modelling.
x
Ignoring the "Duplicates" warning. Duplicate rows are often caused by data pipeline bugs or join errors. They can cause data leakage if they span train and test, and they bias model evaluation.

*
Key takeaways

Summary
Run ydata-profiling first on any new dataset. Use minimal=True for datasets > 100K rows.
The Alerts tab is the most valuable section — it surfaces the most critical data quality issues instantly.
Automated reports flag issues but do not fix them. You still need to act on every warning.
Use sweetviz for train vs test distribution comparison — it immediately flags distribution shift.
Profile only the training set for decisions that affect modelling — profiling the test set causes leakage.
Always document your EDA findings before moving to modelling. What did you find, fix, and what risks remain?
