Data Types in Pandas

The dtype of a column controls its memory use, the operations you can perform, and how ML models interpret it. Getting dtypes right is the first step in any EDA. A column stored as object when it should be category can waste 10× more memory. Dates stored as strings can’t be resampled. Numbers stored as int64 when int8 suffices waste 8× the space.

Common mistakes

✕
Never checking dtypes after loading data. Always run df.dtypes and df.info() right after loading. CSV files store everything as text so pandas has to guess — it often guesses wrong.
✕
Leaving high-cardinality columns as category. Category type only helps when there are few unique values (under ~50% of total). If every row is unique, category uses MORE memory than object.
✕
Forgetting that int columns can't hold NaN. A pandas int64 column becomes float64 if any NaN appears. Use pandas nullable integer type pd.Int64Dtype() if you need both integers and missing values.
✕
Downcasting to int8 without checking the range. int8 holds -128 to 127. If your "age" column has a typo like 999, it silently wraps around to a garbage value. Always check df['col'].max() before downcasting.

★
Key takeaways

Summary
Always run df.dtypes and df.info() first. Fix wrong dtypes before doing anything else.
Use category for string columns with <50 unique values. Can save 10× memory and speed up groupby.
Use int8/int16 for small integers (age, rating, count). Use float32 instead of float64 for ML feature matrices.
Always convert date strings with pd.to_datetime() to unlock .dt accessor methods for feature engineering.
pd.to_numeric(col, downcast='integer') automatically picks the smallest safe integer type.
