KNN Imputation

Instead of filling every NaN with the same global average, KNN imputation finds the K most similar rows (neighbors) and uses their values to fill the gap. Each missing value gets a personalised estimate based on the local structure of your data — far more accurate when features are correlated.

⏱ 22 min ◆ Intermediate sklearn · numpy

◎
The core idea — "ask your neighbours"

Imagine a patient record where Blood Pressure is missing. Simple imputation fills it with the global mean of all patients. KNN imputation instead says: "Find 5 patients who are most similar to this one in age, weight, and cholesterol — then use the average blood pressure of those 5."

This works because features are almost always correlated. A young, light patient likely has different blood pressure than an older, heavier one — the global mean treats them identically. KNN respects local relationships.

Distance metrics — how "similar" is measured

KNN imputation relies on a distance function to find the nearest neighbors. The choice of metric affects which rows are considered similar — and thus what gets imputed. sklearn's KNNImputer uses nan_euclidean_distance by default, which handles missing values in the donor rows gracefully.

⚠

Scale matters — always normalise first! KNN distance treats all features equally. If Age is 0–80 and Income is 0–200,000, the distance is completely dominated by Income. Always apply StandardScaler or MinMaxScaler before KNN imputation — or the imputer will find "nearest neighbors" based on whichever feature has the largest numeric range.

How KNNImputer works — step by step

1
Compute pairwise distances — only using observed features
For any two rows i and j, use only the features that are observed in BOTH rows. Scale the result by n_features / n_observed to compensate for the reduced dimensionality.
d(i, j) = sqrt( (n_features / n_observed) * sum((x_ik - x_jk)^2) ) for k where both x_ik and x_jk are not NaN
2
For each missing cell, find K nearest complete donors
A "donor" row must have a value in the column being imputed (not NaN there). Sort all valid donor rows by their distance to the target row. Take the K closest.
donors = rows where x_j is NOT NaN, sorted by d(target, row) neighbors = donors[:K] # K closest
3
Compute the imputed value (uniform or distance-weighted)
Uniform weights: simple average of the K neighbor values. Distance weights: each neighbor is weighted by 1/distance — closer neighbors contribute more to the fill value.
Uniform: fill = mean(x_j for neighbor in neighbors) Distance: w_i = 1 / d(target, neighbor_i) fill = sum(w_i * x_j_i) / sum(w_i)
4
fit() stores training data; transform() applies it to new data
Unlike SimpleImputer which stores a statistic, KNNImputer stores the entire training set (or uses it as the donor pool). transform() searches this pool for neighbors. Always fit on train only!
imp.fit(X_train) # stores X_train as donor pool X_tr = imp.transform(X_train) # each NaN searches X_train X_te = imp.transform(X_test) # test NaNs also search X_train
5
Time and space complexity
For n training rows, d features, and m missing values in the test set, KNN imputation is significantly more expensive than simple imputation. This matters at scale.
Distance computation: O(n * d) per query row Full imputation: O(n * m * d) — can be slow for large n Memory: O(n * d) — stores entire training set Simple imputation: O(n * d) — just column statistics

Common mistakes

✕
Not scaling features before KNN imputation. Income=50000 vs Age=30 — the Income dimension completely dominates Euclidean distance. Always scale to [0,1] or unit variance first using a Pipeline so the scaler is fit on training data only.
✕
Fitting KNNImputer on the full dataset. Same leakage risk as SimpleImputer. imp.fit(X_all) exposes test statistics to training. Fit on train, transform both — or wrap in Pipeline.
✕
Using KNN on very large datasets without approximation. At n=100k rows, computing pairwise distances is slow. Use sklearn's KNNImputer with a brute-force backend only up to ~10k rows. For larger datasets, consider SimpleImputer + MissingIndicator instead.
✕
Using K=1 (single nearest neighbor). K=1 finds one similar row and copies its value directly — high variance, brittle. If that one row is noisy, you get a bad imputation. Use K≥3 and prefer distance weighting to balance local accuracy with stability.
✕
Ignoring that KNNImputer can't handle rows where all features are NaN. If a row has no observed features, no distance can be computed — there are no valid neighbors. You must drop or handle all-NaN rows before using KNNImputer.

★
Key takeaways

Summary
KNN imputation finds the K most similar rows (by Euclidean distance on observed features) and fills each missing value with their weighted average — far better than a global mean when features are correlated.
Always scale features first. Without scaling, large-magnitude features completely dominate the distance metric, making "nearest neighbor" meaningless. Use a Pipeline: scale → impute.
KNN imputation improves with high feature correlation. When features are independent, it performs no better than the mean. Check feature correlations before choosing this method.
The default metric is nan_euclidean_distance — it automatically handles NaN values in potential donor rows by scaling the distance based on the number of available features.
Use distance weighting (weights='distance') rather than uniform when nearby neighbors are clearly more informative than distant ones. It often improves accuracy at no extra cost.
KNN imputation is O(n²) — for large datasets (>50k rows), consider IterativeImputer (MICE) or falling back to simple imputation + missing indicator columns.
