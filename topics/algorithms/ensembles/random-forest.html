Random Forest

Train hundreds of decision trees on random subsets of data and features, then let them vote. The wisdom of crowds eliminates the high variance of a single tree — giving you accuracy, robustness, and built-in feature importance.

⏱ 25 min ◆ Intermediate numpy · sklearn

🌳
The "wisdom of crowds" idea

A single decision tree is like asking one expert for their opinion — they might be brilliant but also biased by what they've seen. A random forest asks 500 different experts, each trained on different examples and allowed to consider different features. Their collective vote is almost always better than any individual.

The key insight: individual trees make different errors because they're trained on different data (bootstrapping) and consider different features at each split (feature randomness). When you average many uncorrelated errors, they cancel out.

Why not just one big tree? A single deep tree perfectly memorises training data (100% train accuracy, 70% test). Each tree in a forest is intentionally weak (shallow, sees fewer features) — but together they generalise far better. This is the bias-variance tradeoff at work: many high-bias trees combine to get low bias AND low variance.

🎲
Bootstrapping — how each tree gets its data

Each tree trains on a bootstrap sample: randomly sample N points from N training points with replacement. This means ~63.2% unique samples appear (some duplicated), and ~36.8% are never seen — these are the Out-of-Bag (OOB) samples, used as a free validation set.

✓

Free cross-validation! OOB error is nearly as accurate as 5-fold CV with no extra compute. Each tree evaluates its performance on the ~37% of data it never saw. The average OOB error across all trees gives a reliable generalisation estimate — set oob_score=True in sklearn.

✦
Feature randomness — the secret ingredient

At each node, instead of considering all features, each tree only considers a random subset of max_features (default: sqrt(n_features)). This forces diversity — trees can't all split on the same dominant feature, so they learn different patterns.

Feature importance — averaged across all trees

Random Forest feature importance averages the impurity reduction across all trees and all nodes. Because features appear in different positions across different trees, the importance estimate is more robust and less biased than a single tree's importance.

How Random Forest works — step by step

1
Bootstrap: sample data with replacement
For each tree i (i=1..B), draw N samples from training set with replacement. ~63.2% unique points appear; ~36.8% are OOB.
D_i = bootstrap_sample(D, n=N) # with replacement OOB_i = D \ D_i # complement = free validation
2
Train each tree with feature randomness
Build a full/deep CART tree on D_i. At each node, only consider a random subset of m features (m = sqrt(d) for classification, d/3 for regression).
At each node: F_random = random_subset(features, size=m) best_split = argmax_{j in F_random, t} IG(j, t)
3
Aggregate: majority vote (classification) or mean (regression)
Collect predictions from all B trees. Classification: count votes, return majority. Regression: return mean. The aggregation reduces variance by 1/B for uncorrelated trees.
Classification: y_hat = mode({h_1(x), h_2(x), ..., h_B(x)}) Regression: y_hat = (1/B) * sum(h_i(x) for i in 1..B) Var(mean of B uncorrelated estimators) = sigma^2 / B
4
OOB evaluation — no cross-validation needed
For each sample x_j, only trees that didn't see x_j (OOB for that tree) vote on it. Average OOB accuracy is nearly identical to leave-one-out CV accuracy.
OOB_score = mean( 1[y_j == majority_vote({h_i(x_j): j in OOB_i})] for j in 1..N )

Common mistakes

✕
Using too few trees (n_estimators=10). Random forests stabilise around 100–200 trees. With 10 trees, predictions vary significantly between runs. More trees never hurt accuracy — they just cost time. Always check OOB error convergence.
✕
Trusting MDI importance with correlated features. If two features are correlated, importance splits between them randomly — neither looks important. Use permutation importance or SHAP for correlated datasets.
✕
Not using n_jobs=-1. Random forests are embarrassingly parallel — each tree is independent. n_jobs=-1 uses all CPU cores. Training 100 trees with n_jobs=1 vs n_jobs=-1 on an 8-core machine is 8x slower for no reason.
✕
Scaling features before Random Forest. Tree-based models split by threshold comparisons — they're invariant to monotonic transformations. Scaling does absolutely nothing and wastes a pipeline step.
✕
Ignoring class imbalance. Random Forest will predict the majority class. Set class_weight='balanced' or use BalancedRandomForest from imbalanced-learn. Check recall/F1 per class, not just accuracy.

★
Key takeaways

Summary
Random Forest = bagging (bootstrap + averaging) + feature randomness. Both reduce variance without increasing bias.
OOB score is a free, reliable validation metric. Set oob_score=True and check it before running CV.
n_estimators has diminishing returns after ~100. max_features=sqrt(d) is almost always the right default for classification.
Always use n_jobs=-1 — trees are fully independent and parallelise perfectly.
For tabular data with mixed features, Random Forest is the best baseline — it needs no scaling, handles missing values (with imputation), and resists overfitting.
When you need even better accuracy, move to Gradient Boosting (XGBoost/LightGBM). When you need interpretability, stay with Random Forest.
