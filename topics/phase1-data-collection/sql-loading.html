SQL Loading

Most production ML data lives in relational databases — not CSV files. SQL (Structured Query Language) is how you ask a database for exactly the rows and columns you need. Python's sqlite3 (built-in) and SQLAlchemy + pandas make it simple to run a SQL query and get a DataFrame.

One SQL query → one pandas DataFrame. Every time you call pd.read_sql(query, connection), pandas runs your SQL and returns a DataFrame. You can do filtering, joining, grouping, and aggregation in SQL before loading — much faster than loading everything and filtering in pandas.

🔌
Connecting to different databases

The connection string is the only thing that changes between databases. Once connected, pd.read_sql() works identically for SQLite, PostgreSQL, MySQL, and others.

SQLite
sqlite:///mydata.db
File-based, zero setup, built into Python. Perfect for learning and small projects. Supports up to ~100GB files.
PostgreSQL
postgresql://user:pass@host/db
Most popular for production ML. Supports JSON, arrays, full-text search. Install: pip install psycopg2
MySQL / MariaDB
mysql+pymysql://user:pass@host/db
Very common in web applications. Install: pip install PyMySQL
SQL Server
mssql+pyodbc://user:pass@host/db
Common in enterprise environments. Install: pip install pyodbc
BigQuery
bigquery://project/dataset
Google Cloud data warehouse for very large datasets (TB-scale). Install: pip install pandas-gbq
DuckDB
duckdb:///mydata.duckdb
Fast in-process analytical DB. Can query Parquet/CSV directly. Great for data science. Install: pip install duckdb

SQL injection — never build queries with string formatting

SQL injection is a critical security vulnerability. If you put user input directly into a SQL string, an attacker can inject malicious SQL that reads, deletes, or corrupts your database. Always use parameterised queries.

WRONG — string formatting (dangerous)
user_id = "1 OR 1=1"
query = f"SELECT * FROM users
    WHERE id = {user_id}"
# This fetches ALL users!
CORRECT — parameterised query (safe)
user_id = "1 OR 1=1"
query = "SELECT * FROM users
    WHERE id = ?"
df = pd.read_sql(query, conn,
    params=(user_id,))
# Safely returns no rows
⚠

Golden rule: Never use f"...{variable}..." or "...%s" % variable for SQL queries. Always pass values as parameters: pd.read_sql(query, conn, params=(val1, val2)) or use SQLAlchemy's text() with bindparams.

Key takeaways

Summary
Use pd.read_sql(query, connection) to run a SQL query and get a DataFrame in one line.
Filter in SQL, not pandas — a WHERE clause only sends you the rows you need. Fetching a million rows and filtering in Python wastes memory and time.
Always use parameterised queries — never format user input directly into SQL strings.
For learning, use SQLite (built-in). For production, use PostgreSQL. The Python code is almost identical — only the connection string changes.
Use df.to_sql("table_name", conn) to save a DataFrame back to a database — completes the read/write cycle.
