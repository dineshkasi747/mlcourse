API Fetching

Most real-world ML datasets come from APIs — weather data, stock prices, social media feeds, government datasets, sensor readings. An API (Application Programming Interface) is a structured way to request data from a server. You send a request, the server sends back data, usually in JSON format.

How APIs work — request and response

Every API interaction follows the same simple pattern: you send a request (what data you want), the server sends back a response (the data, in JSON format). Think of it like ordering at a restaurant — you specify what you want, the kitchen sends it back.

GET
Fetch data
Most common for ML data collection. Ask the server for data without changing anything. All parameters go in the URL.
POST
Send data
Send a body of data to the server — used for search queries, filtering, or uploading data. Parameters in the request body as JSON.
AUTH
Authenticated requests
Most APIs require a key or token. Pass it as a header: Authorization: Bearer YOUR_KEY. Never hardcode keys in your code.
GET is the simplest method — just build a URL with your parameters and the API returns data. Example: https://api.open-meteo.com/v1/forecast?latitude=51.5&longitude=-0.1&hourly=temperature_2m

Rate limiting — don't get banned

APIs limit how many requests you can make per minute/hour/day. Exceed the limit and you get a 429 Too Many Requests error and possibly a temporary ban. Good ML data collection always includes rate limiting and retry logic.

⚠

Golden rules for API data collection: Always add time.sleep() between requests. Use exponential backoff on errors (wait 1s, then 2s, then 4s, then 8s). Store data as you go — don't fetch everything then save. Respect the API's terms of service.

📄
Pagination — fetching large datasets page by page

APIs rarely return all data at once. They paginate — giving you 100 records at a time with a link to the next page. Your ML pipeline must loop through all pages to get the complete dataset.

Common mistakes

✕
Not checking the status code. Always check response.status_code == 200 before using data. A 404 or 500 response still returns something — but it is an error message, not your data.
✕
Hardcoding API keys in your code. Never put API keys directly in your script. Use environment variables: import os; key = os.getenv("API_KEY"). Keys in code end up on GitHub and get stolen.
✕
No retry logic for failures. Networks fail. APIs have timeouts. Wrap your requests in try/except and retry with exponential backoff. Otherwise one flaky request kills your entire data pipeline.
✕
Fetching all data then saving. If your script crashes halfway through 10,000 API calls, you lose everything. Save data incrementally — write to a file or database after every page.

★
Key takeaways

Summary
APIs send data in JSON format. Use requests.get(url) → response.json() → pd.DataFrame() to get a DataFrame.
Always check response.status_code. 200 = OK, 401 = bad key, 404 = not found, 429 = rate limited, 500 = server error.
Use time.sleep() between requests and exponential backoff for retries. Never hammer an API.
Loop through all pages using while True + a next-page token or offset counter. Save data after every page.
Store API keys in environment variables, never in code.
